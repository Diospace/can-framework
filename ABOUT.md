# About the Can Framework

## Vision
The **Can Framework** aims to bridge the gap between developer productivity and raw browser performance. By leveraging a "Compiler-First" approach, it removes the need for a Virtual DOM, providing a reactive experience that is as fast as hand-optimized JavaScript.

## Technical Excellence

### 1. Fine-Grained Reactivity
We use a signal-based reactivity system. Unlike VDOM frameworks that re-render component trees, Can uses a dependency-tracking engine that targets individual DOM text nodes and attributes.

### 2. The Transpilation Pipeline
Our build process (orchestrated by the CLI) performs several critical steps:
- **SFC Decomposition**: Splitting `.can` files into Script, Template, and Styles.
- **Reactive Binding**: Wrapping template expressions in `effect()` calls.
- **CSS Encapsulation**: Injecting unique scope attributes into the DOM and CSS selectors to prevent style leakage.
- **Module Conversion**: Transforming everything into standard ES Modules (`.mjs`) for native browser support.

### 3. Integrated Ecosystem
Can is "batteries-included." You don't need to hunt for third-party libraries for core needs:
- **Reactivity**: `@can/reactivity`
- **Routing**: `@can/router`
- **State Management**: `@can/store`
- **Server Rendering**: `@can/server-renderer` (SSG/SSR)

## Project Structure

| Directory | Purpose |
|-----------|---------|
| `src/compiler` | The heart of the framework; transforms `.can` to `.mjs`. |
| `src/runtime-core` | Base classes, lifecycle management, and custom element logic. |
| `src/reactivity` | The Signal and Effect engine. |
| `src/cli` | Command-line interface for development and production builds. |
| `ide-extension` | Visual Studio Code language support source code. |

## Why "Can"?
The name "Can" represents **capability**. We believe developers *can* build complex, high-performance applications without the weight of traditional framework runtimes.
